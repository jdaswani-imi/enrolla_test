## Module Overview

[[06_Finance-M08_Finance_Billing|M08]] is the finance and billing engine for Enrolla. It manages invoice creation, payment recording, discounts, fee waivers, credits, refunds, and all financial reporting. [[06_Finance-M08_Finance_Billing|M08]] integrates directly with [[03_Student-M04_Enrolment_Lifecycle|M04]] (enrolment), [[04_Academic-M05_Timetabling_Scheduling|M05]] (session scheduling), [[04_Academic-M11_Academic_Courses|M11]] (catalogue pricing), and [[08_Management-M10_Management_Dashboard|M10]] (management dashboard). All financial data is subject to a 5-year minimum retention requirement under UAE VAT Law.

# 01.1 Admin Billing Authority

Admin can create and issue invoices, apply discounts, apply fee waivers, issue credits, record payments, and perform all billing actions without prior approval from Admin Head. For financial adjustments, Admin must log a reason. The reason is permanently recorded. Admin Head is notified immediately for discounts. Cash refunds require a three-stage approval process: Admin initiates, Admin Head approves, Super Admin approves (see 01.10). Admin can activate the fee-exempt student toggle without prior approval. A logged reason is mandatory.

---

# 01.2 Invoice Creation

Invoices are generated by Admin from the student profile or from the bulk invoice flow. Each invoice line item corresponds to one subject enrolment for one term.

| **Element** | **Detail** |
|---|---|
| Who creates | Admin, Admin Head, Super Admin |
| Line item structure | Subject name, department, year group, session duration, validity (term end date), QTY (session count), unit price (per-session rate after tier discount applied), amount (QTY × unit price) |
| Discount line item | Manual discounts appear as a separate line item below all subject lines, deducted from the subtotal |
| Totals block | Subtotal → manual discount → total excl. VAT → VAT (5%) → rounded off (to nearest fils) → total → credit applied → amount paid → amount due |
| Invoice number | Prefix + sequential number configured in [[09_Settings-M20_Tenant_Settings|M20]]. Format locks 14 days after Go-Live. Numbers are unique across all branches of the organisation. |
| Invoice statuses | Draft, Issued, Part (partially paid), Paid, Overdue, Cancelled, Zero-value |
| Cancellation | Admin cancels with logged reason. Cancelled invoices are retained permanently — never deleted. A Cancelled invoice can generate a replacement invoice. |
| Activity log | Permanent log on every invoice: every action (created, issued, payment recorded, discount applied, cancelled) with actor name, role, timestamp, and reason. |
| Internal note | Staff-only note field on each invoice. Never appears on the PDF sent to parent. |

# 01.3 Enrolment Fee

The AED 300 lifetime enrolment fee is attached to the student record at the point of first real enrolment, not to the first invoice created. This prevents zero-value invoices from consuming the once-per-student fee.

| **Element** | **Detail** |
|---|---|
| Trigger | First invoice where the total before discounts is greater than AED 0 and the invoice is for a subject enrolment. The fee appears as a line item on that invoice. |
| Zero-value invoices | A zero-value invoice created for any reason does not trigger the enrolment fee. The student record flag remains Not Yet Applied. |
| Trial invoices | Trial invoices NEVER trigger the AED 300 enrolment fee. The enrolment fee is attached to the first term invoice generated after a lead converts to an enrolled student. Trial invoices (for Primary: AED 250, Secondary: AED 300) are standalone invoices with no enrolment fee attached. |
| Student record flag | Enrolment Fee status is visible on the student profile: Enrolment Fee Paid (confirmed) or Not Yet Applied (pending). Admin can manually mark as Paid or Waived with a logged reason. |
| Lifetime rule | The enrolment fee is charged once per student, regardless of how many subjects they enrol in over their time at IMI. |

# 01.4 Pricing and Discount Tiers

Pricing is defined in the [[04_Academic-M11_Academic_Courses|M11]] Academic Catalogue per subject. Discount tiers are frequency-based for Secondary students. Primary pricing is flat per year group.

## 01.4.1 IMI Primary Pricing

| **Year Group** | **Rate** |
|---|---|
| KG1–KG2 | AED 160 per session |
| Y1–Y3 | AED 170 per session |
| Y4–Y6 | AED 180 per session |
| Science Y4–Y6 — bundle rate | AED 150 per session when the student is also enrolled in Maths and English with a minimum of 10 sessions each in the same term. Otherwise AED 180. |
| Trial — Primary | AED 250 |

When a student on a bundle rate withdraws from one subject mid-term, the bundle rate remains in effect for the full term — it is not retroactively recalculated. Any refund for the withdrawn subject is reduced by the bundle discount difference: the difference between the bundle rate and the single-subject rate for the sessions remaining. This ensures the student is not credited more than they would have paid without the bundle.

## 01.4.2 IMI Secondary Pricing — Frequency Tiers

| **Year Group** | **Rate** | **Notes** |
|---|---|---|
| Y7–Y9 | 1/week: AED 200 \| 2/week: AED 180 \| 3+/week: AED 160 | Min 10 sessions/subject |
| Y10 | 1–2/week: AED 180 \| 3/week: AED 150 \| 4+/week: AED 125 | Min 10 sessions/subject |
| Y11 | 1–2/week: AED 200 \| 3/week: AED 180 \| 4+/week: AED 150 | Min 10 sessions/subject |
| Y12–Y13 | 1/week: AED 200 \| 2/week: AED 180 \| 3+/week: AED 160 | Min 10 sessions/subject |
| Senior | AED 350 per session | For adult learners and external professionals in the Senior student category |
| Trial — Secondary | AED 300 | One trial per assessment recommendation |
| CAT4 | AED 200 per session | No tier |

## 01.4.3 Tier Calculation Rules

| **Rule** | **Detail** |
|---|---|
| Counting sessions | Total weekly sessions = sum of all subject session frequencies for the student. A subject with 2 sessions per week counts as 2. |
| Uniform rate | When a tier is reached, the tier rate applies to ALL sessions across ALL subjects — not just the session that tipped the tier. |
| Mid-term add | Adding a subject mid-term recalculates the tier from the add date. If the student underpaid (tier increased, rate increased): the system calculates the shortfall and generates a proposed supplementary invoice for the difference only. Admin reviews and confirms before the supplementary invoice is issued — it is never auto-fired. |
| Mid-term remove | Removing a subject mid-term recalculates the tier from the remove date. The original invoice is never modified once any payment is recorded against it (Rule 185). If the student overpaid (tier dropped, rate decreased): the difference is calculated and applied as a credit note on the guardian account automatically. Admin is notified. If the student underpaid: a supplementary invoice for the shortfall is proposed for Admin review. The system always calculates and proposes — Admin always confirms before any financial document is created or issued. |
| Upsell nudge | Invoice builder shows: current tier, next tier threshold, and AED saving if the student adds one more session per week. |

# 01.5 VAT Treatment

| **Element** | **Detail** |
|---|---|
| Rate | 5% — applied to the subtotal after all discounts are applied. VAT is never applied before discounts. |
| TRN | Tax Registration Number appears on all invoices. Required for UAE VAT compliance. |
| Credits (v1) | All credits — including goodwill credits — are applied as post-VAT ledger deductions. The VAT on the original invoice has already been paid. Credits reduce the net amount owed. No VAT credit note is generated in v1. Formal VAT credit note generation via Zoho Books is Phase 2. Admin issues credits with a logged reason. All credits are permanently recorded in the audit trail. |
| Pre-collected credit | Post-VAT deduction. The VAT on the original invoice has already been paid. Credits reduce the net amount owed — they are not subject to additional VAT. |
| Rounding | Invoice total rounded to the nearest fils (0.05 AED). Rounding difference shown as a rounded off line item. |
| Zero-rate and exempt | Tuition fees for students under 18 may qualify for zero-rating under UAE VAT Law. IMI to confirm VAT treatment with their tax adviser. Platform applies the configured rate — the rate configuration is the tenant's responsibility. |

# 01.6 Payment Recording

| **Element** | **Detail** |
|---|---|
| Who records | Admin, Admin Head, Super Admin |
| Payment modes | Cash, bank transfer, cheque, link payment (Phase 2: Telr, Network International, Stripe) |
| Recording a payment | Admin selects invoice, enters amount, payment mode, reference number (bank transfer/cheque), date received, and optional note. Payment is logged permanently. |
| Partial payments | Payment can be recorded below the invoice total. Invoice status changes to Part. Remaining balance is tracked. |
| Below minimum instalment | If the payment falls below 1/n of the invoice total (minimum instalment floor), a warning is shown and enrolment remains Pending. Admin Head must actively approve before enrolment activates. |
| Below configured split | If payment meets the 1/n floor but is below the configured split (e.g. 55% on a 60/40 plan), Admin Head is notified immediately. No approval gate — Admin logs reason and proceeds. |
| Overpayment | If a payment exceeds the invoice total, the excess is applied as a pre-collected credit on the student account. |

# 01.7 Discounts

Discounts are manual reductions applied to an invoice by Admin. They are distinct from frequency-tier pricing, which is automatically applied from the catalogue.

| **Element** | **Detail** |
|---|---|
| Authority | Admin applies discounts without prior approval. Admin must log a reason. |
| Admin Head notification | Admin Head receives an immediate in-app notification when any discount is applied: student name, invoice, discount amount, reason, Admin name. |
| VAT | Discount is applied before VAT calculation. VAT is applied to the discounted subtotal. |
| Discount line item | Manual discount appears as a separate line on the invoice PDF below all subject lines. |
| 100% discount — nudge | If a manual discount causes the invoice total to reach AED 0, a confirmation nudge appears before saving: "This discount brings the invoice total to AED 0. This invoice will auto-resolve to Paid. Are you sure?" Admin Head is notified immediately on confirmation. |

Only one discount can be applied per invoice line item. Discounts do not stack. The discount field accepts either a fixed AED amount or a percentage reduction — only one discount applies per line. Admin applies discounts without prior approval.

---

# 01.8 Fee Waivers

Fee waivers are explicit exemptions applied by Admin without prior approval. They are distinct from discounts in that they waive a specific fee type rather than reducing the invoice total by a percentage or amount.

| **Element** | **Detail** |
|---|---|
| Trial fee waiver | Admin waives the trial class fee at the point of invoice creation. Logged reason required. No prior approval. |
| Session fee waiver | Admin waives the session fee for one or more sessions on an invoice. Logged reason required. No prior approval. |
| Enrolment fee waiver | Admin waives the AED 300 enrolment fee. The student record flag updates to Waived. Logged reason required. No prior approval. |
| Fee-exempt student toggle | Admin can toggle a student as fully fee-exempt. No prior approval required. A logged reason is mandatory. All invoice line items are built at full value and then zeroed. Zero-value invoices auto-resolve to Paid with no parent notification. |
| All waivers | Permanently logged in the invoice activity log and the audit trail with the waiving Admin's name, timestamp, and reason. |

# 01.9 Credits

| **Element** | **Detail** |
|---|---|
| Pre-collected credit | Issued when a parent has overpaid or paid for sessions that were not delivered. Post-VAT deduction — the original VAT has already been paid. Admin issues with logged reason. No Admin Head approval required. Post-VAT deduction. No VAT credit note generated in v1. |
| Goodwill credit | Issued as a gesture of goodwill. Admin issues with logged reason. No Admin Head approval required. Post-VAT ledger deduction. No VAT credit note generated in v1. Zoho Books VAT credit note generation is Phase 2. |
| Credit application | Credits are applied to future invoices for the same student. Credit balance is visible on the student profile. |
| Authority | Admin issues all credit types with a logged reason. No prior approval required. |

Credits are stored as an internal ledger balance on the guardian account. A credit is not a separate invoice type and does not carry a separate credit note number series. A printable credit note PDF can be generated on demand from the guardian profile at any time — the PDF is generated when requested, not automatically on credit creation. The credit note PDF contains the following line items: (1) Credit amount (net — the AED value of the credit before VAT); (2) VAT credited (5% of the net credit amount — this reduces the VAT liability for the relevant tax period); (3) Total credit (gross — net plus VAT credited). The VAT period field auto-fills to the current UAE tax period (the quarter in which the credit is being issued). Admin can override the VAT period before generating the PDF — for example when a credit relates to a prior period. The platform pre-fills the current period as the default to minimise error.

# 01.10 Session Transfer Between Siblings

Session transfer allows unused session credits from one student's invoice to be applied to a sibling's account where both students are linked under the same guardian. Session transfer is disabled at IMI (toggle Off in [[09_Settings-M20_Tenant_Settings|M20]]) but is available to tenants who enable it.

| **Element** | **Detail** |
|---|---|
| Eligibility | Both students must be linked as siblings under the same guardian account (confirmed Sibling Group in [[03_Student-M02_Student_Guardian_CRM|M02]]). |
| Who requests | Admin initiates the transfer request from the source student's invoice. |
| Approval | Admin Head approval required. Admin Head receives an in-app approval request. |
| What transfers | A whole-number session count only. Partial sessions cannot be transferred. No AED amount moves — only session units. |
| Financial record | The source invoice is updated to reflect the transferred sessions. The receiving student's invoice is updated to reflect the additional sessions. Both updates are logged in the activity log of both invoices with the Admin and Admin Head names, timestamp, reason, and session count transferred. |
| VAT treatment | No VAT adjustment on transfer. Session units move between invoices at face value. |
| Reversal | Session transfer cannot be reversed once Admin Head has approved. Admin Head can reject the request before approving, which cancels it with no changes made. |
| Toggle | Off by default. Enabled per tenant in [[09_Settings-M20_Tenant_Settings|M20]] Feature Toggles. |
| IMI status | Toggle Off — not used at IMI. |

# 01.11 Refunds

Refunds are the return of money already paid by a parent. They require three-stage approval before any funds are released. No funds leave the business without Super Admin sign-off.

All refund types (cash, bank transfer, credit reversal) follow the same 3-stage approval chain: (1) Admin initiates the refund request with reason; (2) Admin Head approves; (3) Super Admin approves. No refund is processed until all three stages are complete.

| **Element** | **Detail** |
|---|---|
| Default outcome | Credit is the default outcome when a student overpays or is owed money after a mid-term change. A refund is only processed at explicit parent request. |
| Stage 1 — Admin raises request | Admin raises a refund request from the invoice or student profile. Logged reason required. Refund amount and method (cash, bank transfer, credit reversal) specified. Status: Pending Admin Head. |
| Stage 2 — Admin Head approval | Admin Head reviews and approves or rejects. On approval, status advances to Pending Super Admin. On rejection, Admin is notified and the refund does not proceed. |
| Stage 3 — Super Admin hard gate | Super Admin must explicitly approve before execution. Hard block — the refund cannot be processed until Super Admin approves. On approval, status becomes Approved. |
| HR release notification | On Super Admin approval, HR/Finance receives an immediate in-app notification: "Refund of AED [X] approved for [Student name]. Release within 14 days." The 14-day release window is the default. Super Admin can set a shorter window. |
| Self-approval prevention | The requester and any approver at any stage cannot be the same user ID. If Admin Head submitted the original request, they cannot approve at Stage 2. |
| Audit trail | Every stage of the refund process is permanently logged: who requested, who approved at each stage, timestamps, amounts, reasons. |

# 01.12 Zero-Value Invoices

Zero-value invoices are legitimate in many scenarios: fully fee-exempt students, 100% scholarship, adjustments, or vouchered sessions. They auto-resolve to Paid with no parent notification.

| **Element** | **Detail** |
|---|---|
| Creation | Admin creates a zero-value invoice like any other invoice. No Admin Head approval required. |
| Auto-resolution | Zero-value invoices auto-resolve to Paid immediately on issue. No payment recording step required. |
| Fee-exempt toggle | Invoices generated for fee-exempt students are always zero-value. Line items are built at full value then zeroed — the financial structure is preserved for reporting. |
| 100% discount nudge | If a manual discount causes an invoice to reach AED 0, a confirmation nudge is shown before saving. Admin Head is notified on confirmation. |
| Enrolment fee trigger | Zero-value invoices do not trigger the enrolment fee. |
| Reporting | Zero-value invoices are included in all financial reports. They are not hidden or excluded. Zero-value invoices generated via discount are clearly distinguishable from fee-exempt invoices in reporting. |

# 01.13 Unbilled Sessions

Sessions attended without a corresponding active invoice are logged in the Unbilled Sessions tracker. They can be retrospectively billed once an invoice is generated.

| **Element** | **Detail** |
|---|---|
| Unbilled Sessions tracker | Accessible from Admin and Admin Head dashboards. Shows all sessions attended with no invoice: student, subject, date, number of unbilled sessions. |
| Auto-task | On confirmation of an unbilled attendance, a high-priority task is raised for Admin: "Invoice required — [Student name] attended [Subject] on [date] with no active invoice." |
| Admin Head alert | Admin Head receives an immediate in-app alert on any unbilled attendance confirmation. |
| Retrospective deduction | Admin generates an invoice for the student and subject. During invoice creation, the system shows unbilled sessions for that student/subject. Admin selects sessions to include. On first payment, session deductions apply retrospectively. |
| Write-off | If unbilled sessions are not to be invoiced, Admin marks them as resolved in the tracker with a logged reason (Waived / Error / Compensatory). The sessions are removed from the tracker but remain in the audit log. |

# 01.14 Bad Debt

| **Element** | **Detail** |
|---|---|
| Trigger | When a student withdraws mid-term and the invoice recalculation shows an outstanding balance that is unlikely to be collected, Admin Head can mark the balance as bad debt. |
| Authority | Admin Head marks bad debt with a logged reason. Super Admin is notified. The balance is written off on the ledger. |
| VAT treatment | Bad debt write-offs do not automatically adjust the VAT liability on the original invoice. See VAT note below. Zoho Books sync in Phase 2. |
| Record retention | Bad debt records are retained for 5 years minimum. The original invoice and all payment history are retained. |

> **VAT note:** Marking a balance as bad debt does not automatically adjust the VAT liability recorded on the original invoice. Under UAE VAT Law, bad debt relief rules may apply. Admin Head must confirm with the organisation's tax adviser whether a VAT adjustment is required before the write-off is finalised. The platform records the write-off permanently but does not alter the original VAT record.

# 01.17 Data Retention

All financial records are subject to a 5-year minimum retention requirement under UAE VAT Law (Federal Decree-Law No. 8 of 2017). This is non-negotiable and cannot be shortened by any user action including Super Admin.

| **Rule** | **Detail** |
|---|---|
| Minimum retention | 5 years from the invoice date or the end of the relevant tax period, whichever is later |
| Deletion | Not possible for any financial record within the 5-year window. Cancellation is the terminal action. Cancelled records are retained. |
| Post-retention | After 5 years, records may be archived or deleted at the tenant's discretion. Enrolla does not auto-delete. Manual action by Super Admin. |
| PDPL interaction | If a data subject submits a PDPL erasure request, financial records within the 5-year window are retained. PII is anonymised. Financial record structure (amounts, dates, line items) is retained with anonymised student ID. See [[01_Foundation-PL03_Data_Privacy|PL-03]] for full erasure workflow. |

# 01.18 Role-Based Access

Finance module access is tightly restricted. Academic Head, HOD, Teacher, and TA have no `finance.view` — the Finance nav item is hidden entirely for these roles. All finance actions listed below are hidden for unauthorised roles, never greyed out.

| **Role** | **Finance access** |
|---|---|
| Super Admin | Full access — all finance actions across all branches. Three-stage refund: final approver. |
| Admin Head | Full access — all finance actions within branch. Three-stage refund: stage 2 approver. Receives immediate notification on all discounts applied. |
| Admin | Create, edit, void invoices. Record payments. Apply discounts (with logged reason, Admin Head notified). Issue credits. Initiate refunds (stage 1). Apply fee waivers. Mark bad debt. Bulk invoice generation. |
| HR/Finance | Full finance view. Create, edit, void invoices. Record payments. Apply discounts. Request discounts. Issue credits. Request refunds. Waive fees. Mark bad debt. Export financial data. View salary data. Cannot approve or final-approve refunds. |
| Academic Head | No finance access — `finance.view` not granted. Finance nav item hidden. |
| HOD | No finance access — `finance.view` not granted. Finance nav item hidden. |
| Teacher | No finance access. |
| TA | No finance access. |

**Key boundaries:**
- Refund final approval (`finance.finalApproveRefund`) — Super Admin only.
- Refund stage 2 approval (`finance.approveRefund`) — Admin Head and Super Admin.
- Salary data (`finance.viewSalary`) — Super Admin and HR/Finance only.
- Bulk invoice generation (`bulk.generate.invoices`) — Super Admin, Admin Head, Admin only.
